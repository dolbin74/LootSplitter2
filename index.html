REFLECTION

Reflection: How My Phase 2 System Works In my application, the loot array is the single source of truth. Every piece of information the interface displays—such as the list of items, the total loot value, and the split results—comes directly from the data stored in that array. The screen never stores its own numbers. Instead, the interface always reflects whatever is currently inside the array. When the array changes, the UI changes. When the array is empty, the UI shows an empty‑state message. When the array has items, the UI renders them in the loot table. I added quantity to my data model because Phase 2 requires total loot to be calculated as value × quantity for each item. This also makes the system more realistic: many items in a game or dungeon scenario come in stacks. By storing quantity inside each object, the array can represent the true state of the loot, and updateUI() can calculate totals correctly from that state. The updateUI() function is the center of the entire system. It performs all calculations and all rendering in one place. Whenever the state changes—whether loot is added, loot is removed, or the party size changes—updateUI() recalculates the total loot, rebuilds the loot table, updates the split button’s enabled/disabled state, and shows or hides the results. No other function is allowed to calculate totals or update the screen. This keeps the system predictable and prevents duplicate logic. The state controls what is visible or enabled. For example, when the loot array is empty, updateUI() shows the “No loot added” message, hides the split results, and disables the Split button. When the party size is invalid, the Split button is disabled even if loot exists. When both the loot array and party size are valid, the Split button automatically enables. The interface always responds to the current state instead of button clicks. Finally, conditional logic protects the system from invalid state. Before adding loot, the app checks that the name is not empty, the value is a valid non‑negative number, and the quantity is at least 1. Before splitting loot, the app checks that the party size is valid and that loot exists. These checks prevent bad data from entering the array and ensure that updateUI() always works with clean, valid state.
